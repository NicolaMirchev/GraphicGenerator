About the project Click here to read about the current state and idea for the project.

The idea of the project came after first week in university and the struggle to find the best combination for attending all the courses that I had been assigned for. The complexity came, because for each course there are few groups(in different time), which can be visited.

Also, many times the personal schedule, given to you from university assign two lectures(courses) in a single hour (e.g "Data structures" and "Data bases" at 13:30 on Monday). So i decided to exercise with the technology I had been learning in this moment (Spring Boot), the ability to present the entities in the best possible way for the computer and to choose best data structures and algorithms, so the project could give you the BEST SCEDULE (with shortest gaps), based on given courses and options when to attend them.

What the program does is generating all possible combinations for attending the given courses and after that chooses the best option for the user(at this moment there are two options - less gaps, but more days at the univirsity, or bigger gap some day/s, but less days at university).

Used technologies Click here to read about the technologies used for building this project.

The first idea was only to exercise with Sping Boot and Spring Data JPA, so the back-end is Spring application, consisting of different packages- for cotrollers(REST API), services (where the algorithms about the combinations are executed) and percistance, where are the repositories, related to the data base. For development proccess i used postgres docker container. I made unit tests for each and after each service algorithm implementation, to be sure they work,

I wanted to make some UI, where anyone could check his scedule and decided to exercise with Angular, because I had previous experience. I decided that for the "first release" the project will be simpler and DB connection won't be necessary and the whole requiered data will be collected from the browser and sended at once through request. The application procceed the data in the service layer and return a result which is based on the client browser collected data.

Optimization

The entity model is more complex than the model used for this release and the algorithms use the complex model. This means that there is a conversion from the one model to the other, before executing the algorithms, which slows down the proccess. The one solution that I have in mind is make an algorithms more generic and abstract. Another is migrate the model, when there are more complex features, so the more complex model could be used.
